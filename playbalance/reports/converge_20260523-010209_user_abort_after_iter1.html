해석 / 메모

💬 이번 세션 대화 요약

왜 이 KPI 였나

사용자가 평균 생존시간 3분~4분30초 (180~270s) 를 목표로 설정. cause=death 조건은 timer cap outlier 자동 제외 및 σ 안정성 동시 강제.

사전 가설

  • spawn_burst_count 미세조정 (Ghost Count): 직전 pass 사이클의 입증 lever
  • enemy_hp 약화: kill snowball 가속화
  • level_up_xp_increment 감소: leveling 가속, snowball 보조

사용자가 표명한 우려

  • enemy_damage 정수 round-trip 흡수 (직전 abort 재현됨)
  • spawn_burst_count derived target 이 plan 의도 (Ghost) 와 어긋날 가능성
  • KPI gap +96s 대비 lever 수와 강도 부족

※ 이 박스는 Claude가 종료 시점에 대화 로그를 요약한 재구성입니다. 정확한 사용자 발언은 대화 원문을 참조하세요.

이번 사이클이 입증한 것

  • level_up_xp_increment 단독 lever 는 이 게임의 Dur 에 무영향 (적어도 -19% 한 사이클 분량)
  • spawn_burst_count knob 의 derived target window 가 실제 bottleneck (Ghost burst) 이 아닌 modal spawn (Gear/FlySkull, Random 패턴) 을 잡음 — config 설계 한계
  • bal-apply --auto 진단계가 발화하는 knob 매칭이 KPI 의도와 어긋날 수 있음 (이번엔 plan 의 Ghost lever 의도 → 실제로는 Gear no-op 적용)

다음 세션 후속 가설

  • Option 1 — Ghost.asset 외 7개 변종 Count 를 unity-cli 로 수동 직접 조정 (Count 15→18 자연사망 유도 / 15→11 더 약화). bal-apply --auto 우회. plan 의 원래 의도 직접 구현.
  • Option 2 — bal-apply config 에 spawn_burst_count_ghost 같은 별도 knob 신설: asset_glob 을 Ghost 폴더로 좁히고 derived target window 제거 (항상 Ghost 변종 적용).
  • Option 3 — spawn_burst_count 의 asset_match 에 'pattern_filter' 추가: Dense/Circle 패턴 자산만 매칭 (Random 자동 제외). 게임-비종속성 유지하면서 trap 회피.
  • Option 4 — KPI 점진적 접근 'Dur 100~150 AND cause=death' 부터 시도. 현 baseline 84s 와 가까워 도달 가능. 한 단계씩 KPI 올리기.
  • Option 5 — v18 재학습: 모델이 inference 환경 (Ghost Count=15) 에 학습된 적 없음. policy mismatch 누적이 다음 사이클부터 더 악화. 학습 모드 fixedDeltaTime 도 동시 실측 (이전 hand-off 의 ⚠ 미해결).
  • Option 6 — 진단계 추가 finding: 'kpi_distance_high' 메타-finding 신설. KPI 거리 N 이상이면 모든 knob 후보 노출. reactive bottleneck 한계 해소.
  • 조치 우선순위 추천: Option 1 (수동 Ghost 조정) 먼저 → Dur 효과 즉시 측정 → 효과 입증되면 Option 3 (config 정교화) 으로 자동화 복귀.
