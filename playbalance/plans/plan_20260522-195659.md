## 2. Direction

### KPI
`Dur >= 90`

이유: baseline 62s 대비 90s 도달만 해도 burst 축소 효과 검증으로 충분. KPI 180s 같은 큰 목표는 시그널 보기 전엔 무의미.

### Baseline
자동 (`baseline=auto`) — bal-converge 가 직전 session 재사용 또는 새 측정.

### Knob whitelist
**spawn_burst_count 단일** (사용자 요청).

이미 적용된 분리 (`.claude/bal-apply.json` 편집 완료):
- `spawn_rate.triggers = ["spawn_cliff"]` — `structural_duration_cutoff` 제거됨
- `spawn_burst_count.triggers = ["structural_duration_cutoff"]` — 신규
- `spawn_intensity_curve.triggers = ["spawn_cliff", "structural_duration_cutoff"]` — manual 액션 (auto knob 과 충돌 없음)

→ `structural_duration_cutoff` 진단 시 spawn_burst_count 가 단독 후보. 별도 임시 편집 불필요.

다만 다른 knob (enemy_damage, enemy_hp, level_up_xp_increment, player_max_hp) 들은 자기 diagnostic 에는 여전히 반응. **`Dur >= 90` 진단이 structural_duration_cutoff 하나로만 분류되는지** 가 관건. 다른 diagnostic 도 같이 fire 되면 다른 knob 도 활성화될 수 있음 — 발생 시 plan 위반은 아니나 결과 해석 시 분리해서 봐야 함.

## 4. Traps (주의)

- **derived target = Ghost 일 것** (cycle 2 와 동일 패턴). bal-apply 가 `event.enemy_spawn.set_key` 의 사망 5초 전 mode 로 Ghost 를 찾아낼 것. 만약 derived target 이 Gear 로 잘못 잡히면 Gear.asset (Pattern=Random) 의 Count 를 만지게 되는데 Random 은 Count 무시 → no-op. 결과 비대칭이면 derived target 로그 확인.
- **Count=15 round-trip 안전**: 정수 30 의 -50% = 15 (clean). 더 작은 자산값 (Bat.asset Count=1 같은 것) 에는 -50% = 0.5 → 1 또는 0 round 발생 위험. Ghost 만 매칭되면 OK.
- **iter 1 play_no=1 누락**: 이전 두 cycle 모두 첫 play 가 짧게 단명 (11~42s). logger ShutdownAsync 의심. 통계 N=5 가 사실상 N=4 일 수 있음 — 결과 해석 시 chart 의 첫 play 길이 확인.
- **60s cutoff 가 안 뚫리면 두 가지 해석**: (a) 가설 자체 오류, (b) Ghost 외 다른 burst (Gear, FlySkull 등) 가 같은 시점에 함께 죽임. 코드상 60s 변종이 Ghost 만이라고 확인됐으나 실측이 우선.
